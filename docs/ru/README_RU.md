# langembed — Русская документация

> **English documentation:** [README.md](../../README.md)

## Что это такое

Цель исследования — обучить sentence embeddings (векторные представления предложений) с нуля для языка с ограниченными ресурсами (в качестве примера используется гуджарати). Проект реализует три параллельных ветки обучения:

- **Ветка A (с нуля)**: собственный BPE-токенизатор + небольшой энкодер в стиле RoBERTa + контрастивное обучение SimCSE. Подход не требует готовых многоязычных моделей и полностью контролируем.
- **Ветка B (multilingual)**: многоязычная модель (mBERT или XLM-R), дообученная на тех же данных от носителей языка. Позволяет сравнить, насколько трансфер из большой модели превосходит обучение с нуля.
- **Ветка C (LLM)**: декодерная языковая модель (LLM) используется как эмбеддер через LoRA (подход llm2vec/LoRA). Веса LLM замораживаются, дообучаются только LoRA-адаптеры.

Носители языка занимают центральное место в пайплайне: они обеспечивают сигнал разметки и контроль качества через цикл активного обучения (active learning loop). Система подаёт аннотаторам наиболее «неопределённые» пары предложений, тем самым максимизируя информационную ценность каждой размеченной пары.

Целевая метрика качества — коэффициент ранговой корреляции Спирмена на изолированном тестовом наборе STS (Semantic Textual Similarity). Тестовый набор никогда не используется при обучении — это гарантируется архитектурным инвариантом.

## Архитектура

Пайплайн состоит из восьми фаз (0–7) плюс дополнительной фазы 4C:

- **Фаза 0** — нормализация и дедупликация (`preprocess.normalize`): единственный источник истины для всей обработки текста.
- **Фаза 1** — сбор корпуса (`build_corpus.py`): агрегация и дедупликация сырых текстовых файлов.
- **Фаза 2** — обучение BPE-токенизатора (`train_tokenizer.py`): построение словаря на основе корпуса.
- **Фаза 3** — MLM-претрейн энкодера (`train_mlm.py`): предобучение RoBERTa-подобного энкодера на задаче маскированного языкового моделирования.
- **Фаза 4** — SimCSE — контрастивное обучение (`train_simcse.py`): улучшение качества эмбеддингов через контрастивный loss.
- **Фаза 4C** — LLM как эмбеддер через LoRA (`llm_embed/`): LoRA-адаптация декодерной LLM для получения высококачественных эмбеддингов.
- **Фаза 5** — сервис разметки с active learning (`annotation/`): FastAPI-приложение для сбора разметки от носителей языка.
- **Фаза 6** — оценка (`evaluate.py`): Spearman-корреляция на STS, Recall@k и MRR@k.
- **Фаза 7** — HTTP-сервис `/embed` (`serving/serve.py`): эндпоинт для инференса в продакшн.

Три архитектурных инварианта:

1. **Единая нормализация** — функция `preprocess.normalize` является единственным источником истины; текст нормализуется только через неё, без дублирования логики.
2. **Запрет утечки теста** — файлы `data/sts_test_*` никогда не попадают в обучающие данные ни на одной фазе.
3. **Конфиги через YAML** — все гиперпараметры задаются в файлах `configs/*.yaml`; магические числа в коде обучения запрещены.

Поток данных: **корпус → токенизатор → энкодер → контрастивное обучение → цикл разметки → оценка → сервинг**.

## Структура репозитория

```
src/langembed/
  config.py          — загрузка YAML-конфигов (load_config)
  preprocess.py      — нормализация текста (единственный источник истины)
  data/
    build_corpus.py  — сбор и дедупликация корпуса
  tokenizer/
    train_tokenizer.py — BPE-токенизатор (HuggingFace tokenizers)
  pretrain/
    train_mlm.py     — MLM-претрейн на базе RoBERTa-конфигурации
  contrastive/
    train_simcse.py  — SimCSE через SentenceTransformers
  llm_embed/
    train_lora.py    — LoRA-адаптация LLM для получения эмбеддингов
  annotation/
    api.py           — FastAPI-сервис: /queue, /annotate, /export
    db.py, models.py — SQLAlchemy: PostgreSQL + Redis
  eval/
    evaluate.py      — Spearman, Recall@k, MRR@k
  serving/
    serve.py         — FastAPI /embed эндпоинт

configs/             — YAML-гиперпараметры для каждой фазы
tests/               — unit, API и e2e тесты
docs/                — документация (en + ru)
Makefile             — удобные команды: make lint, make test, make test-e2e
docker-compose.yml   — postgres, redis, annotation, serve, train
dvc.yaml             — DAG воспроизводимости
```

## Установка

**Локальная установка:**

```bash
git clone <repo>
cd langembed
pip install -e ".[ml,serve,dev]"
cp .env.example .env  # задать пароли
```

**Через Docker:**

```bash
docker compose up -d postgres redis
docker build --target base -t langembed-annotation .
docker build --target ml -t langembed-ml .
docker compose up -d
```

## Запуск пайплайна

```bash
# 1. Сырые данные
cp your_data.txt data/raw/wiki_gu.txt

# 2. Инфраструктура
docker compose up -d postgres redis

# 3. Полный пайплайн
dvc repro

# 4. Оценка
make eval

# 5. Просмотр результатов
mlflow ui
```

Для быстрой проверки отдельных фаз используйте smoke-тесты:

```bash
make pretrain-smoke   # smoke-тест фазы MLM-претрейна
make simcse-smoke     # smoke-тест фазы SimCSE
```

## Фазы 0–7 + 4C

**Фаза 0 — нормализация.** Применяет NFKC-нормализацию Unicode, нормализацию IndicNLP (для гуджарати) и схлопывание множественных пробелов. Функция `preprocess.normalize` вызывается автоматически во всех последующих фазах и является единственным местом, где производится нормализация текста.

**Фаза 1 — сбор корпуса.** Читает все файлы из `data/raw/*.txt`, применяет MinHash-дедупликацию для удаления дублирующихся предложений и записывает результат в `data/corpus.txt`. Конфигурируется через `configs/corpus.yaml` (параметры: `raw_paths`, `out_path`, `threshold`).

**Фаза 2 — токенизатор.** Обучает BPE-токенизатор с Whitespace-претокенизатором и NFKC-нормализатором, размер словаря по умолчанию — 8000 токенов. Артефакт сохраняется в `artifacts/tokenizer/`. Конфиг: `configs/tokenizer.yaml`.

**Фаза 3 — MLM-претрейн.** Обучает RoBERTa-подобный энкодер на задаче маскированного языкового моделирования с использованием HuggingFace Trainer. Архитектурные параметры (`hidden_size`, `num_layers` и др.) задаются в `configs/pretrain.yaml`. Метрики (loss, perplexity) логируются в MLflow. Артефакт: `artifacts/encoder/`.

**Фаза 4 — SimCSE.** Контрастивное обучение на собственных предложениях: позитивные пары получаются двумя прямыми проходами одного предложения через энкодер с разными dropout-масками. Реализована через обёртку SentenceTransformer. Артефакт: `artifacts/simcse/`. Конфиг: `configs/simcse.yaml`.

**Фаза 4C — LLM LoRA.** LoRA-адаптация декодерной языковой модели в стиле LLaMA для получения sentence embeddings через mean-pooling по последнему скрытому слою. Замораживаются все веса базовой модели, дообучаются только LoRA-адаптеры. Артефакт: `artifacts/llm_lora/`. Конфиг: `configs/llm_lora.yaml`.

**Фаза 5 — сервис разметки.** FastAPI-приложение на порту 8001 для сбора разметки от носителей языка. PostgreSQL хранит пары предложений и метки, Redis используется как брокер Celery. Алгоритм active learning выбирает пары с максимальной неопределённостью: `1 - |cos - 0.5| / 0.5`.

**Фаза 6 — оценка.** Вычисляет Spearman-корреляцию между предсказанным косинусным сходством и человеческими оценками на STS-тесте, а также Recall@k и MRR@k для задачи поиска. Результаты записываются в `metrics/eval.json` и в MLflow.

**Фаза 7 — сервинг.** FastAPI-приложение на порту 8000 с эндпоинтом `/embed`, который принимает список строк и возвращает L2-нормализованные векторы в формате JSON. Модель загружается из директории, указанной в переменной окружения `LANGEMBED_MODEL_DIR`.

## Сервис разметки и active learning

Сервис разметки (`annotation/api.py`) предоставляет три эндпоинта:

- **`/queue?annotator_id=N&n=10`** — возвращает N наиболее неопределённых пар предложений из очереди разметки плюс 2 «золотых» калибровочных вопроса с заранее известными ответами. Золотые вопросы позволяют выявить недобросовестных аннотаторов.
- **`/annotate`** — сохраняет оценку аннотатора по шкале от 0 до 5 для конкретной пары предложений.
- **`/export`** — агрегирует метки нескольких аннотаторов и формирует триплеты (якорь, позитив, негатив) для supervised-обучения.

**Формула неопределённости**: `1 - |cos - 0.5| / 0.5`. Максимальная неопределённость достигается при косинусном сходстве cos = 0.5 (предложения похожи ровно наполовину — аннотатор с наибольшей вероятностью даст информативную метку). При cos = 0 или cos = 1 неопределённость равна нулю — пара очевидна.

**Качество аннотаторов** отслеживается через взвешенную каппу Коэна (weighted kappa). Метки от разных аннотаторов агрегируются с весами, пропорциональными их надёжности (reliability-weighted aggregation). Аннотаторы, систематически расходящиеся с консенсусом, получают меньший вес.

Конфигурации Label Studio для инициализации проектов разметки хранятся в `annotation/label_studio/`. Для загрузки первичных золотых пар используется скрипт `scripts/seed_gold.py` — он наполняет таблицу золотых вопросов до начала основной кампании разметки.

## Тестирование

Тестовое покрытие организовано в три слоя:

1. **Unit-тесты** (чистые функции): проверяют `normalize`, `dedup`, `uncertainty_from_cosine`, `weighted_kappa`, `aggregate` в изоляции без внешних зависимостей. Запускаются командой `make test`.

2. **API contract-тесты**: проверяют контракты эндпоинтов сервиса разметки (`/queue`, `/annotate`, `/export`) и эндпоинта `/embed` с использованием FastAPI TestClient и SQLite в памяти (StaticPool). Позволяют тестировать API без запущенных PostgreSQL и Redis. Запускаются командой `make test`.

3. **E2E smoke-тест**: полный пайплайн на английском языке с использованием CPU — `build_corpus` → `tokenizer` → MLM (50 шагов) → SimCSE (1 эпоха) → `evaluate`. Запускается командой `make test-e2e`.

После 50 шагов MLM-претрейна качество модели (Spearman) ожидаемо будет низким — e2e тест проверяет работоспособность кодового пути, а не качество модели.

```bash
make test          # unit + API тесты
make test-e2e      # полный пайплайн на английском (несколько минут)
make lint          # ruff + mypy
```

## DVC и воспроизводимость

Команда `dvc repro` запускает граф воспроизводимости (DAG), описанный в `dvc.yaml`. Стадии выполняются в порядке зависимостей: **corpus → tokenizer → pretrain → simcse → supervised → llm_lora → evaluate**.

```bash
dvc repro            # запустить все устаревшие стадии
dvc metrics show     # показать метрики из eval.json
dvc dag              # визуализировать DAG
```

Веса моделей и данные отслеживаются DVC и исключены из git (`.gitignore`, `.dvcignore`). Для командной работы необходимо настроить удалённое хранилище DVC:

```bash
dvc remote add -d myremote s3://my-bucket/langembed
dvc push
```

Это гарантирует, что любой член команды сможет воспроизвести эксперимент с теми же данными и весами, выполнив `dvc pull && dvc repro`.

## MLflow

MLM-претрейн логирует loss и perplexity в MLflow после каждого шага оценки. Фаза оценки записывает Spearman-корреляцию, Recall@k и MRR@k как в `metrics/eval.json`, так и в MLflow.

```bash
mlflow ui          # открыть http://localhost:5000
```

Для сравнения прогонов: **Experiments → выбрать эксперимент → отметить нужные прогоны → Compare**. Рекомендуется сравнивать три прогона: ветка A (с нуля), ветка B (multilingual fine-tuned), ветка C (LLM LoRA). Визуализация позволяет определить, какая архитектура даёт лучшую Spearman-корреляцию при данном объёме обучающих данных.

## Адаптация под другой язык

Чтобы переключиться с гуджарати на другой язык, выполните следующие шаги:

1. **`configs/tokenizer.yaml`**: установите `language: <lang_code>` и обновите `raw_paths` на пути к корпусу нового языка.
2. **`configs/eval.yaml`**: обновите `test_path` на путь к STS-тесту для нового языка.
3. **`preprocess.normalize`**: параметр `lang` в `_indic_normalizer` управляется конфигом (по умолчанию `"gu"`). Для неиндийских языков нормализатор IndicNLP автоматически пропускается, применяются только NFC-нормализация Unicode и схлопывание пробелов.
4. Запустите `dvc repro` — все стадии параметризованы через конфиги и будут перезапущены автоматически.

Примечание: для языков, не входящих в семью индийских языков, библиотека `indic-nlp-library` пропускается автоматически — дополнительных изменений кода не требуется.
